# コーディング規約

scrapbook の開発において参考とする設計原則・コーディング規約をまとめる。
厳格に遵守することを求めるわけではないが、コードの整合性や一貫性を保つためのガードレールとして参照してもらいたい。

- 『単体テストの考え方/使い方』（Vladimir Khorikov）
- 『良いコード/悪いコードで学ぶ設計入門』（仙塲大也）
- 『体系的に学ぶ 安全な Web アプリケーションの作り方 第2版』（徳丸浩）

## 1. アーキテクチャ

> 出典: 『単体テストの考え方/使い方』

### レイヤー構成

`domain`（ドメイン層）と `usecase`（ユースケース層）を中心に構成し、外部依存は `infra` 層に隔離する。

| 層 | 責務 | 依存先 |
| --- | --- | --- |
| domain | ビジネスルール、値オブジェクト、エンティティ | なし（純粋） |
| usecase | アプリケーション固有のワークフロー | domain |
| infra | DB 接続、外部 API、Queue 操作 | domain, usecase |

### 依存の方向

依存は常に **外側から内側へ** 向かう。domain 層は他の層を知らない。

``` md
infra → usecase → domain
```

usecase は infra の具体実装を知らず、interface（Repository 等）を通じてやり取りする。これにより DB やサービスの差し替え、単体テストが容易になる。

### 関数型アーキテクチャの考え方

『単体テストの考え方/使い方』で紹介される functional core / mutable shell の考え方を取り入れる。

- **functional core（domain）**: 副作用を持たない純粋なロジック。入力から出力を計算するだけ
- **mutable shell（infra / usecase の一部）**: DB操作や外部呼び出しなど副作用を伴う処理

domain層のロジックを純粋に保つことで、テストしやすく、変更に強い構造を維持する。

## 2. 設計原則

> 出典: 『良いコード/悪いコードで学ぶ設計入門』、『単体テストの考え方/使い方』

### 高凝集・疎結合

- 関連するデータとロジックは同じモジュールにまとめる（高凝集）
- モジュール間の依存は最小限にする（疎結合）
- データだけを持つクラスとロジックだけを持つクラスに分離しない

### 不変性の優先

- 変数は原則として再代入しない（`const`、`readonly` を使う）
- オブジェクトの状態を変更するのではなく、新しいオブジェクトを返す
- 可変が必要な場合は、そのスコープを最小限に閉じ込める

### 完全コンストラクタ

- オブジェクトの生成時にすべての必須フィールドを初期化する
- ガード節で不正な値を弾き、不完全な状態のオブジェクト（生焼けオブジェクト）を作らせない
- 当プロジェクトにおいては完全コンストラクタの考え方は単一エンティティの生成時に適用する。関連エンティティの非同期充実とは別の話なので注意

### 値オブジェクト

- 金額、URL、タグ名など、意味のある値はプリミティブ型のまま扱わず、専用の型として表現する
- バリデーションロジックを型の内部に閉じ込めることで、不正な値の流通を防ぐ
- ただし、適用範囲はトレードオフを意識する。すべてのプリミティブを値オブジェクト化すると可読性が下がる場合がある

### 早期リターン

- 異常系は関数の先頭で弾き、正常系のネストを浅く保つ

### 命名

- 具体的で意味範囲が狭い名前を選ぶ
- 存在ベース（`Manager`、`Processor`）ではなく目的ベースで命名する
- `Common`、`Util`、`Helper` のような汎用的な名前は避ける
- 省略や連番を使わない

### 継承よりコンポジション

- 継承は密結合を生みやすい。共通の振る舞いはコンポジションで実現する

### DRY（Don't Repeat Yourself）

- 同じ知識やロジックを複数箇所に書かない。変更理由が同じコードが2箇所以上にあるなら共通化を検討する
- ただし、見た目が似ているだけで変更理由が異なるコードを無理に共通化しない。誤った共通化は結合度を上げる

### YAGNI（You Aren't Gonna Need It）

- 現時点で必要のない機能やインターフェースを先回りして作らない
- 「将来使うかもしれない」は実装の根拠にならない。必要になった時点で追加する

### KISS（Keep It Simple, Stupid）

- 解決策は可能な限りシンプルに保つ。同じ目的を達成できるなら、より単純な方法を選ぶ
- 抽象化やパターンの導入は、それによって実際の複雑さが減る場合にのみ行う

### +@

DRY,YAGNI,KISSを基本としつつ、下記の点についても注意を払いなさい。

- エラーハンドリングは適切か
- エッジケースは考慮されているか。
- 既存のパターンと一貫性はあるか
- パフォーマンス上の懸念点はないか

## 3. テスト

> 出典: 『単体テストの考え方/使い方』

### テストの目的

テストの目的は「ソフトウェア開発プロジェクトの成長を持続可能にする」こと。価値をもたらさないテストは負債であり、作成すべきではない。

### テストの 4 本柱

良いテストは以下の 4 要素の掛け算で評価する。

1. **退行に対する保護**: バグを検出できるか
2. **リファクタリングへの耐性**: 実装を変えたとき偽陽性が出ないか
3. **迅速なフィードバック**: 速く実行できるか
4. **保守のしやすさ**: テスト自体が理解・修正しやすいか

### テスト対象の優先順位

| 複雑さ / 重要性 | 協力オブジェクト少 | 協力オブジェクト多 |
| --- | --- | --- |
| 高 | **domain（最優先でテスト）** | 過度に複雑なコード（分割すべき） |
| 低 | 取るに足らないコード（テスト不要） | controller / infra |

- domain 層のロジックに対する単体テストが最も費用対効果が高い
- 過度に複雑なコードは、ロジック部分を domain に抽出して分離する

### テスト手法の優先順位

1. **出力値ベーステスト**（最も推奨）: 入力に対する戻り値を検証する。副作用がなく、リファクタリング耐性が高い
2. **状態ベーステスト**: 操作後のオブジェクトの状態を検証する
3. **コミュニケーションベーステスト**（最後の手段）: モックで呼び出しを検証する。偽陽性のリスクが高い

### モックの使用基準

- **使ってよい場合**: 外部システムとの通信（API 呼び出し、メール送信など）で、かつ外部から観察可能な振る舞い
- **使ってはいけない場合**: システム内部のオブジェクト間通信（実装詳細）

### テストで避けるべきこと

- プライベートメソッドを直接テストしない
- テストのためにプロダクションコードの可視性を変えない
- 実装詳細ではなく、最終的に得られる結果を検証する

## 4. セキュリティ

> 出典: 『体系的に学ぶ 安全な Web アプリケーションの作り方 第2版』

### 基本方針

セキュリティ対策は特別な追加作業ではなく、日常的な開発プロセスの一部として組み込む。

### インジェクション対策の共通原則

すべてのインジェクション脆弱性（SQL インジェクション、XSS、コマンドインジェクション等）は同じ構造を持つ。ユーザー入力が、埋め込み先の仕様においてコードとして解釈されることが根本原因。

対策は **検証とエスケープの両方** を行う。どちらか一方では不十分。

- **検証（入力側）**: ホワイトリスト型で、想定する仕様に適合する入力のみを受け入れる
- **エスケープ（出力側）**: 埋め込み先の仕様に合わせた文字変換を行う。ライブラリや公式関数を使う

### 主要な脆弱性と対策

| 脆弱性 | 対策 |
| --- | --- |
| SQL インジェクション | プレースホルダ（パラメータ化クエリ）を使う。文字列結合で SQL を組み立てない |
| XSS | HTML 出力時にエスケープする。フレームワークの自動エスケープ機能を活用する |
| CSRF | トークンによるリクエスト検証。SameSite Cookie 属性の設定 |
| CORS | 許可するオリジンを明示的に指定する。`*` を安易に使わない |

### scrapbook 固有の留意点

- Browser Extension からの保存リクエストは外部入力として扱い、バリデーションを行う
- URL のメタ情報取得時、取得先のコンテンツを信頼しない（OGP タグの内容をそのまま表示しない）
- AI 生成結果（タグ、要約等）も UI に表示する前にエスケープする

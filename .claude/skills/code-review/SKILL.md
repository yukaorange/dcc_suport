# コードレビュー

task: 現在のdiffが、プロジェクトのコーディングガイドラインに沿っているかを3観点から検証し、ユーザにフィードバックを与え、指摘を反映する。

拡張思考を行いなさい。
**ultrathink**

claude codeのAgent Teams機能を使って、以下の3つのチームメイトで並列レビューを実施する。

## チーム構成

### チームメイトA: コーディング規約チェック

- 参照: `/.claude/rules/coderule.md`
- 観点: RULE-001〜013への準拠（関数型スタイル、命名規則、class禁止、try/catch制限、早期リターン、switch/if使い分け、Reactパターン等）

### チームメイトB: 設計・アーキテクチャチェック

- 参照: `/.claude/rules/convention.md` セクション1, 2
- 観点: レイヤー構成と依存方向、functional core / mutable shell、高凝集・疎結合、不変性、完全コンストラクタ、値オブジェクト、DRY/YAGNI/KISS

### チームメイトC: セキュリティ・パフォーマンスチェック

- 参照: `/.claude/rules/convention.md` セクション3, 4
- 観点: インジェクション対策、XSS/CSRF/CORS、scrapbook固有の留意点（Extension入力のバリデーション、OGPコンテンツの非信頼、AI生成結果のエスケープ）、パフォーマンス上の懸念

## ワークフロー

### STEP 1

`git diff main` を実行する。

### STEP 2

各チームメイトがSTEP1のdiffに対し、担当観点のルールファイルを参照して違反点を検討する。
必要があれば対象ファイルのコード全体を読み込んでもよい。

NEVER: この時点ではtypecheck/lintは不要

### STEP 3

全チームメイトの検出結果を統合し、変更提案にFB-連番(FB-1, FB-2...)をつけてユーザに簡潔に提示する。

- 各FBに観点ラベルを付与する: `[規約]` `[設計]` `[セキュリティ]`
- 対象ファイルパス、対象コードの引用と、変更後のコードをそれぞれ含める。

そのうえで、ユーザに変更提案に関してのフィードバックまたは実装許可を求める。

### STEP 4

STEP3で許可された変更をファイルに適用する。
NEVER: ユーザのフィードバックで明示的に拒否された指摘点はファイルに反映を行わない。ï

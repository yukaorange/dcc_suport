# Task

拡張思考を行いなさい
**ultrathink**

コンテキスト７のMCPを使用し、ドキュメントを調べて正確に検討しなさい。
**use Context7**

提供されたIssueの対応方針を策定し、コードを書き始めるのに十分な実装プランを作成する。

claude codeのAgent teams機能を使って、以下の3つのチームメイトで並列にプランを策定する。

## チーム構成

### チームメイトA: Frontend

- 担当: UI/UX、Reactコンポーネント設計、状態管理、ルーティング
- 参照: `/.claude/rules/coderule.md`（RULE-009〜012等React関連）、`/.claude/rules/convention.md`
- 成果物: フロントエンドの実装ステップ、変更対象ファイルと変更後コードの具体案
- Backendチームメイトに対して、必要なAPIエンドポイント・リクエスト/レスポンスの型を明示すること

### チームメイトB: Backend(API) / Infra

- 担当: APIエンドポイント設計、domain/usecase/infraのレイヤー構成、DB操作、外部サービス連携
- 参照: `/.claude/rules/convention.md`（セクション1: アーキテクチャ、セクション2: 設計原則、セクション4: セキュリティ）
- 成果物: バックエンドの実装ステップ、変更対象ファイルと変更後コードの具体案
- Frontendチームメイトに対して、提供するAPIの仕様（エンドポイント、メソッド、型）を明示すること

### チームメイトC: クロスレイヤー整合性チェック

- 担当: A・Bのプランを突き合わせ、レイヤー間の矛盾や抜けを検出する
- 検出すべき問題:
  - Frontendが期待するAPIとBackendが提供するAPIの不一致（エンドポイント、型、レスポンス構造）
  - データフローの抜け（Frontend→Backend→DBの流れに欠落がないか）
  - エラーハンドリングの不整合（Backendが返すエラーをFrontendが処理できるか）
  - セキュリティ上の懸念（バリデーションの責務がどちらにも属していない等）
- 成果物: 整合性の問題リストと修正提案

## ワークフロー

### STEP 1

`gh issue view $ARGUMENTS --json body --template '{{.body}}'`

を実行して、issue本文を取得する。

### STEP 2

STEP1の内容に関連しそうなファイルをすべて読み込む。
STEP1を実装するに足りると確信できるまでコードベース理解を続ける。

### STEP 3

チームメイトA・Bがそれぞれの担当レイヤーで詳細な実装プランを策定する。
チームメイトCがA・Bのプランを突き合わせ、整合性の問題を検出する。

コンテキスト７のMCPを使用し、ドキュメントを調べて正確に検討しなさい。 **use Context7**
context7でカバーできない範囲にあってはにwebで最新のドキュメントなどを調査しなさい。

実装プランの書き方:

- 簡潔な日本語
- 今回の対応を完遂するためのステップを分解し、ステップごとに記載
  - ステップは具体的であるべき
  - それぞれ、既存コードのファイルパスと、変更後のコードの具体案を含める
- チームメイトCの指摘がある場合は、A・Bのプランに反映した上で統合する

CAUTION:

- 既存のコードベースに関しては推測を含めず、STEP2で取得した実際のコードのみを信頼する
- 判断に迷うことはユーザに判断を仰ぐ
- ライブラリ導入などを伴うプランの場合、context7のmcpサーバーを使って最新のドキュメントを確認し、破綻のない提案を出すこと

### STEP 4

統合した実装プランを`.work/plan-${ISSUE_ID}.md`に書き込む。

プランファイルの構成:

```md
# SCR-XX: [issueタイトル]

## 概要
[issueの要約]

## Frontend実装ステップ
[チームメイトAのプラン]

## Backend実装ステップ
[チームメイトBのプラン]

## 整合性チェック結果
[チームメイトCの検出結果と反映状況]
```

codexレビュー:

実装プランの案について、codex commandで計画のレビューを受けること。具体的な使い方は以下に記す。
レビュー指示の文章は適宜調整すること。ただし、codexは本質的じゃない指摘もすることがあるので、「瑣末な点への不要なリプライはするな。致命的な点への指摘に尽力せよ。」という指示は入れた方がいい。
codexの指摘がなくなるまでアップデート→レビューを繰り返すこと（なお、甲乙つけ難い選択肢など、ユーザーの判断が必要な項目がある場合は、あなたの意見を付した上でSTEP5で判断を仰ぐ扱いとし、ループを終了して良い。）

codexを呼ぶときはtimeoutを延長すること。(7分以上を推奨)

``` bash
# initial plan review request
# 必ず-mでモデルを指定すること（2027.2時点ではgpt-5.3-codexが最適）
codex exec -m gpt-5.3-codex "このプランをレビューして。瑣末な点への不要なリプライはするな。致命的な点への指摘に尽力せよ。: {plan_full_path} (ref: CLAUDE.md, .claude/rules/coderule.md, .claude/rules/convention.md)"

# updated plan review request
# resume --last をつけないと最初のレビューのコンテキストを忘却されるので注意
codex exec resume --last -m gpt-5.3-codex "プランを変更したからレビューして。瑣末な点への不要なリプライはするな。致命的な点への指摘に尽力せよ。: {plan_full_path} (ref: CLAUDE.md, .claude/rules/coderule.md, .claude/rules/convention.md)"

```

レビューを経た最終産物について自身でもセルフレビューし、codexの指摘を受け入れたことで破綻がないかを見直すこと。
納得のいく出来になったらプラン完成として良い。

### STEP 5

プランの概要をユーザに提示し、質疑応答を行う。
